# Stage 4 — Market Data Service (Rust)

## Goal

Consume `book-updates-eq` and `trades-eq` from Kafka, maintain an
in-memory book mirror, and serve L1/L2 data to WebSocket subscribers.

**Current simplification:** The C++ engine publishes full L2 snapshots.
The Rust mirror simply replaces on each update — no delta accumulation
needed. If the engine later switches to incremental deltas (only changed
levels), the Rust side would need to accumulate them to reconstruct full
state. Not on the roadmap; documented as a future optimization.

## Data Model

```rust
struct Level { price: u64, quantity: u64 }

struct OrderBookMirror {
    bids: Vec<Level>,       // high-to-low
    asks: Vec<Level>,       // low-to-high
    last_trade: Option<Trade>,
}

impl OrderBookMirror {
    fn apply_book(&mut self, update: BookUpdate)  // replace bids/asks
    fn apply_trade(&mut self, trade: TradeEvent)  // update last_trade
    fn l1(&self) -> Option<L1Snapshot>
    fn l2(&self) -> L2Snapshot
}

struct L1Snapshot {
    best_bid: Level,
    best_ask: Level,
    last: Option<Trade>,
}
```

**Current simplification:** `apply_book` replaces bids/asks entirely. If the
engine switches to incremental deltas, `apply_book` would accumulate deltas
instead. The rest of the system (L1/L2 queries, WebSocket broadcast) stays
unchanged.

## Kafka Consumer

| Topic             | Schema       | Updates                      |
| ----------------- | ------------ | ---------------------------- |
| `book-updates-eq` | `BookUpdate` | Full L2 snapshot → replace   |
| `trades-eq`       | `TradeEvent` | Last trade → `last_trade` L1 |

Single consumer, both topics, same consumer group. FlatBuffers
deserialization via `flatbuffers` crate (generated with `flatc --rust`).

**NBBO:** The best bid and best ask (first entry of each side) are served
implicitly as part of L1. No separate NBBO feed is needed in this
single-venue setup.

## WebSocket Protocol

```
→ {"type": "subscribe", "level": "l1"}   // or "l2"
← {"type": "l1", "bid": {p, q}, "ask": {p, q}, "last": {p, q, seq}}
← {"type": "l2", "bids": [...], "asks": [...], "last": ...}
```

**Snapshot on connect + stream forward:** Since deltas are full snapshots,
every update is self-contained. New subscriber gets current mirror, then
the next broadcast message is already a complete state. No offset tracking
needed.

## Concurrency

```
Kafka consumer → RwLock<Mirror> → broadcast::Sender<Update> → per-connection tasks
```

The consumer holds the write lock briefly per update. WebSocket connections
hold the read lock only during snapshot serialization on connect. Subsequent
updates arrive via `tokio::sync::broadcast` — no lock contention on the hot
path.

## Milestones

### 1. Scaffold + Kafka Consume

- `cargo init` at `rs/`
- Add dependencies
- Compile `.fbs` schemas with `flatc --rust`
- Consume `book-updates-eq` and `trades-eq`
- Deserialize FlatBuffers, update `OrderBookMirror`
- Log L1 on every update to verify

### 2. L1 WebSocket

- Tokio TCP listener on `:8081`
- Accept connections, receive `subscribe {"level": "l1"}`
- Send current L1 snapshot on subscribe
- Forward subsequent book/trade updates as L1 deltas
- Clean up on disconnect

### 3. L2 WebSocket

- Same server, same port — dispatch by subscription level
- Send full book depth on subscribe (L2 snapshot)
- Forward subsequent updates as L2 state
- Graceful handling of slow clients (backpressure, disconnect)

### 4. Graceful Shutdown

- SIGINT/SIGTERM handler
- Close Kafka consumer, commit offsets
- Broadcast close frame to subscribers
- Wait for pending writes, exit

## Out of Scope

- Multiple symbols (single `eq` for now, same as engine)
- Persistence / recovery
- Authentication / TLS
- L3 data (engine doesn't publish individual orders)
- Metrics / OTEL (Stage 6)
- Incremental book deltas (future optimization if engine changes)
